Go 1.11 Modules

译文

Go1.11 中引入了之前提到的版本模块(versioned modules)。modules在Go 1.11版本中是实验的性质，计划在Go 1.13中引入最终版本。尽管还有很多细节会改变，但未来版本都会支持Go1.11 或Go1.12中定义的模块。

2018年1月提出了原型vgo。2018年7月 模块登录主仓库。Go1.1在2018正式发布。

目录内容

快速开始

先看一下如何使用，后面再看一下细节。

  1. 在GOPATH目录外创建一个目录

    $ mkdir -p /tmp/scratchpad/hello
    $ cd /tmp/scratchpad/hello

  2. 初始化一个新的模块

    go mod init github.com/you/hello

  3. 编写代码

    $ cat < hello.go
    package main

    import (

    "fmt"
"rsc.io/quote"

    )

    func main() {

    fmt.Println(quote.Hello())

    }
    EOF

  4. 执行

    $ go build
    $ ./hello

go.mod文件会自动更新，明确指定你的依赖的版本，v1.5.2是一个semver 标记:

cat go.mod
module github.com/you/hello
require rsc.io/quote v1.5.2

日常工作方式

在上面的例子中不需要使用 go get。
你的日常工作方式如下：

  • 在 .go代码中添加你的语句。
  • 标准命令如go build 或 go test会自动添加所需要的依赖，并更新go.mod 并下载新的依赖。
  • 如果有需要，可以通过go get foo@v1.2.3，go get foo@master, go get foo@e3702bed2，或者直接更改go.mod

可能用到的一些功能：

  • go list -m all
  • go list -u -m all
  • go get -u 或 go get -u=patch
  • go build ./... or go test ./...
  • go mod tidy
  • replace
  • go mod vendor

New Concepts

Modules

Module是相关go package的集合，将这些package聚合在一起作为一个独立单元。

Module记录精确的依赖，并创建可复用的构建。

通常，一个版本控制的仓库在其根目录准确的包含一个模块。当然在一个仓库中也支持多个模块。

总结仓库、模块、包的关系：

  • A repository contains one or more Go modules.
  • Each module contains one or more Go packages.
  • Each package consists of one or more Go source files in a single directory

模块必须是根据semver进行语义版本化的，其格式是v(major).(minor).(patch)。比如v0.1.0,v1.2.3,v1.5.0-rc.1。 开头的v是必须的。 如果使用Git，tag就是版本。

go.mod

模块是由go.mod文件和go 源码的文件树定义的。 模块源码需要位于GOPATH目录之外。有四个指令：module,require,replace,exclude.

下面是一个定义模块github.com/my/thing的go.mod文件的例子，

module github.com/my/thing

require (
    github.com/some/dependency v1.2.3
    github.com/another/dependency/v4 v4.0.0
)

通过module指令表名模块的身份，并提供了module path。引入模块内所有包的话，都使用包的path 作为前缀。 模块路径和包相对路径决定了包的import path。

例如，创建了模块github.com/my/repo，模块由两个包组成 github.com/my/repo/foo 和 github.com/my/repo/bar组成。go.mod第一行通常声明模块路径，即 module github.com/my/repo，对应到硬盘的目录结构是：

repo/
├── go.mod
├── bar
│   └── bar.go
└── foo
    └── foo.go

在Go源码中，使用包含模块路径的全路径引入包。举例，go.mod文件中声明模块路径是example.com/my/module，一个消费方应该是这样使用:

import "example.com/my/module/mypkg

这就是从模块example.com/my/module中引入包mypkg的做法。

版本选择(version selection)

  • 引入go.mod中不存在的包时，go build和go test命令都会自动找到合适的模块 和 最大版本的依赖，写入到go.mod文件 作为一个require指令。
  • 一个构建中，所有模块的版本选择使用最小版本选择(minimal version selection)算法。对于构造中的每个模块而言，使用的是require指令中明确列出的语义最大化的版本。

举个例子，如果你的模块依赖模块A 需要D v1.0.0，你的模块同时依赖模块B 需要D v1.1.1，那么根据最小版本选择，那么D v1.1.1会被选择用于构建。即使D v1.2.0可用了，也依然使用D v1.1.1。

最小版本选择算法的简短介绍可以看High Fidelity Builds 或者看 more detailed vgo blog series

语义版本化（Semantic Import Versioning)

这么多年以来，Go官方FAQ中关于包版本有这些建议：

"Packages intended for public use should try to maintain backwards compatibility as they evolve. The Go 1 compatibility guidelines are a good reference here: don't remove exported names, encourage tagged composite literals, and so on. If different functionality is required, add a new name instead of changing an old one. If a complete break is required, create a new package with a new import path."

最后一句很重要：如果破坏了兼容性，你就要更改包的引用路径。 在Go1.11 模块中，这些建议被写入import compatibility rule中：

"If an old package and a new package have the same import path, the new package must be backwards compatible with the old package."

基于Semantic Importing Versioning，Go模块中遵循下面规则：

  • 遵从semver
  • 如果模块是v2或更高版本，那么引用路径要加上版本号。如 github.com/my/mode/v2/
  • 如果版本是v0或v1，则不需要在引用路径上加上版本。

linux同步机制

日常学习和工作中经常用到linux同步机制，因此进行一个总结。

一、为什么需要同步

很简单，同一个资源的修改只能同时被一个进程或线程修改，不然会出现异常情况。

二、同步的机制

  1. 原子操作
  2. 自旋锁
  3. 信号量
  4. 互斥锁
  5. 读写锁

三、自旋锁

自旋锁是linux内核中使用非常频繁的锁，而我们在写c++或golang代码时基本不会直接使用自旋锁。 但是我们遇到性能问题时，通过perf top去查看时经常会看到spin_lock占了非常多的CPU，这是因为很多锁的底层实现都会使用自旋锁。

所谓自旋锁：

当进程A申请锁成功后，进程B申请锁就会失败，但是进程B不会调度，原地自旋。就在原地转到天昏地暗只为等到进程A释放锁。由于不会睡眠和调度的特性，在中断上下文中，数据的保护一般都是选择自旋锁。

自旋锁的实现：

linux中针对每一个spin lock会有两个计数。分别是next和owner（初始值为0）。进程A申请锁时，会判断next和owner的值是否相等。如果相等就代表锁可以申请成功，否则原地自旋。直到owner和next的值相等才会退出自旋。假设进程A申请锁成功，然后会next加1。此时owner值为0，next值为1。进程B也申请锁，保存next得值到局部变量tmp（tmp = 1）中。由于next和owner值不相等，因此原地自旋读取owner的值，判断owner和tmp是否相等，直到相等退出自旋状态。当然next的值还是加1，变成2。进程A释放锁，此时会将owner的值加1，那么此时B进程的owner和tmp的值都是1，因此B进程获得锁。当B进程释放锁后，同样会将owner的值加1。最后owner和next都等于2，代表没有进程持有锁。next就是一个记录申请锁的次数，而owner是持有锁进程的计数值。

共享数据如果是被进程上下文使用，可以使用信号量和互斥量。
共享数据如果是被中断上下文使用，则必须使用自旋锁。 因为在中断上下文中不能使用睡眠的锁。 信号量和互斥量都是睡眠锁。

Spinlock的特点

我们可以总结spin lock的特点如下：

（1）spin lock是一种死等的锁机制。当发生访问资源冲突的时候，可以有两个选择：一个是死等，一个是挂起当前进程，调度其他进程执行。spin lock是一种死等的机制，当前的执行thread会不断的重新尝试直到获取锁进入临界区。

（2）只允许一个thread进入。semaphore可以允许多个thread进入，spin lock不行，一次只能有一个thread获取锁并进入临界区，其他的thread都是在门口不断的尝试。

（3）执行时间短。由于spin lock死等这种特性，因此它使用在那些代码不是非常复杂的临界区（当然也不能太简单，否则使用原子操作或者其他适用简单场景的同步机制就OK了），如果临界区执行时间太长，那么不断在临界区门口“死等”的那些thread是多么的浪费CPU啊（当然，现代CPU的设计都会考虑同步原语的实现，例如ARM提供了WFE和SEV这样的类似指令，避免CPU进入busy loop的悲惨境地）

（4）可以在中断上下文执行。由于不睡眠，因此spin lock可以在中断上下文中适用

场景分析

线程A访问共享资源R，线程B访问共享资源R；

  1. 没有任何锁机制，
    当线程A正在访问R 进行写操作时，线程B访问R进行写操作，同一个资源的同时写操作，会导致程序执行不正确。

  2. 加入自旋锁
    线程A 先成功获得自旋锁，对R进行写操作。 此时B不能获得自旋锁，则一直在等待，线程B不能做其他事情，只是在等待自旋锁。

如果线程A在对R进行写操作时，现在发生中断并执行的是线程B，那么线程B等待自旋锁会一直在等待；而A由于发生中断又不会释放锁，因此出现死锁 。

为了解决这样的问题，linux kernel采用了这样的办法：如果涉及到中断上下文的访问，spin lock需要和禁止本CPU上的中断联合使用。

四、信号量

信号量（semaphore）是进程间通信处理同步互斥的机制。是在多线程环境下使用的一种措施，它负责协调各个进程，以保证他们能够正确、合理的使用公共资源。 它和spin lock最大的不同之处就是：无法获取信号量的进程可以睡眠，因此会导致系统调度。

信号量实现：

一个count计数：标记当前可用资源数量
一个链表：记录等待的进程

五、读写锁

不管是自旋锁、信号量、还是互斥锁 在同一时间只能有一个进程进入临界区。对于有些情况，我们是可以区分读写操作的。因此，我们希望对于读操作的进程可以并发进行。对于写操作只限于一个进程进入临界区。而这种同步机制就是读写锁。读写锁一般具有以下几种性质：

  • 同一时间有且仅有一个写进程进入临界区。
  • 在没有写进程进入临界区的时候，同时可以有多个读进程进入临界区。
  • 读进程和写进程不可以同时进入临界区。

对于某个给定资源的共享访问也称为共享-独占（shared-exclusive)上锁，因为获取一个读写锁用于读称为共享锁（shared lock)；获取一个读写锁用于写称为独占锁（exclusive lock）。

六、互斥锁

前文提到的semaphore在初始化count计数的时候，可以分为计数信号量和互斥信号量（二值信号量）。mutex和初始化计数为1的二值信号量有很大的相似之处。他们都可以用做资源互斥。但是mutex却有一个特殊的地方：只有持锁者才能解锁。但是，二值信号量却可以在一个进程中获取信号量，在另一个进程中释放信号量。如果是应用在嵌入式应用的RTOS，针对mutex的实现还会考虑优先级反转问题。

互斥锁指代相互排斥(mutual exclusion)，是最基本的同步形式，用于保护临界区(critical region)，以保证任何时刻只有一个线程在执行其中代码。保护一个临界区的通常轮廓是：

lock_the_mutex(…)
临界区
unlock_the_mutex(…)

互斥锁保护的是临界区，实际上保护的是在临界区中被操纵的数据（data）。也就是说，互斥锁通常用于保护由多个线程分享的共享数据（shared data）

6.1 生产者和消费者问题

在UNP 第7章中讲述了典型的生产者和消费者的问题。

场景一：生产者全部生产以后，消费者再消费

UNP给出的实现是多生产者单消费者的方式，因此生产者需要对buff加互斥锁。而消费者只需要顺序消费buff中内容即可。

场景二：边生产边消费

那么消费者所在线程会有两种情况：一种是buff内有数据，直接消费；第二种是buff内没有数据，那么消费者线程就必须一直等待，直到有数据到来，才开始消费。

UNP中给出了一个实现（UNP中给出的是一个函数，我是直接写的）：

while(true) {
lock_the_mutex(..)
if (buf.empty) {
unlock_the_mutex(…)
continue
} else {
unlock_the_mutex(…)
break
}
}
operate buf data

上面例子中，当期待的数据没有准备好时，我们的做法是：一次次循环，每次给互斥锁解锁、上锁。这称为轮询（polling），是一种对CPU时间的浪费。

当然我们也许可以睡眠很短的一段时间，但是不知道应该睡眠多久，这儿需要的是另外一种类型的同步，它允许一个线程（或进程）睡眠到发生某个事件为止。

七、条件变量

互斥锁用于上锁，条件变量用于等待。

那么在消费者进程中实现可以变为：

pthread_mutex_lock(mutex_)
while (ready == 0) {
pthread_cond_wait(cond,mutex)
}
pthread_mutex_unlock(mutex_)

在生产者进程中实现变为

pthread_mutex_lock(mutex_)
pthread_cond_signal(cond_)
pthread_mutex_unlock(mutex_)

go版thrift读写超时时间处理

在golang的thrift调用中，如果我们设置的是
`
socket,err := thrift.NewTSocket(host)

defer socket.Close()

socket.SetTimeout(time.Duration(3) * time.Millisecond)
`

这种设置的其实是连接超时、读超时、写超时分别是3毫秒。为什么这样讲呢?

在socket.go中看func (p *TSocket) Write(buf []byte) (int,error) { 中 p.pushDeadline(false, true) 中看到 p.conn.SetWriteDeadline(t)
可以看到读写的超时时间都是socket.SetTimeout中设置的。

在func (p *TSocket) Open() error{中可以到DialTimeout的连接超时也是socket.SetTimeout中设置的。

因此如果要想实现thrift的连接、读、写超时，那么就必须修改socket.go中的pushDeadline，分别为读写设置超时时间。

但如果去更改vendor/git.apache.org/thrift.git/lib/go/thrift/socket.go文件，那么会造成很难维护，那么我们能否在我们代码里面实现socket.go 而不影响原有的包呢?

答案是可以的。

首先看一段thrift调用的代码:
`
socket,err := thrift.NewTSocket(host)

defer socket.Close()

socket.SetTimeout(time.Duration(3) * time.Millisecond)

tf_transport := transportFactory.GetTransport(socket)

defer tf_transport.Close()

err = socket.Open()

…
`
那么从这儿来看，只要我们自己实现TSocket即可。

当然了，我们看一下TSocket到底是什么呢？

二、thrift的核心概念
在文章thrift介绍中给出了thrift的整体结构。

我们知道thrift有几大概念：

  1. 数据类型

  2. 协议protocol
    2.1 TBinaryProtocol 二进制编码格式进行数据传输
    2.2 TCompactProtocol 高效率的、密集的二进制编码格式进行数据传输
    2.3 TJSONProtocol —— 使用 JSON 的数据编码协议进行数据传输
    2.4 TSimpleJSONProtocol —— 只提供 JSON 只写的协议，适用于通过脚本语言解析

  3. 传输层
    3.1 TSocket —— 使用阻塞式 I/O 进行传输，是最常见的模式
    3.2 TFramedTransport —— 使用非阻塞方式，按块的大小进行传输。若使用 TFramedTransport 传输层，其服务器必须修改为非阻塞的服务类型。
    3.3 TNonblockingTransport —— 使用非阻塞方式，用于构建异步客户端

  4. 服务器端类型
    4.1 TSimpleServer —— 单线程服务器端使用标准的阻塞式 I/O
    4.2 TThreadPoolServer —— 多线程服务器端使用标准的阻塞式 I/O
    4.3 TNonblockingServer —— 多线程服务器端使用非阻塞式 I/O

  3. TSocket和TFramedTransport关系

字符编码和字符集

计算机底层使用二进制数来存储和传输数据；
人类使用文字、数字、图形来表达信息；
因此二者之间需要进行某种转换。
文字、数字等字符转换为二进制数被称为编码
二进制数转换为文字、数字称为解码。
文字、数字、图形总称为字符(character)。在golang里面使用rune表示字符。

字符的集合就是字符集(charset)，常见的字符集包含ASCII，GB2312，通用字符集（UCS）和Unicode字符集

ASCII码 使用一个字节表示，第一个bit是0，用7个bit来表示，只能表示128个字符。
GB2312是针对汉字的字符集，使用两个字节来表示；但如果每个国家都有自己的字符集，那字符集太多了，有没有一种字符集可以包括所有字符呢？

Unicode：囊括所有字符，但只是字符集，没有规定字符是如何存储和传输的。 而UTF(unicode transformation format)规定了字符如何传输和存储的。常用的是UTF-8。

UTF-8是变长的编码方式：

  • 对于单字节的符号，字节的第一位置0，后面7位为这个符号的Unicode码。因此，对于英语字母，UTF-8编码和ASCII码是相同的
  • 对于n个字节的字符（n>1）,第一个字节的前n位都置为1，第n+1位置为0，后面字节的前两位一律置为10。剩下的没有提及的二进制位，全部为这个符号的Unicode码

UTF8和unicode之间转换

这里只介绍Unicode转换为UTF-8 ，以汉字 “柯” 为例。已知 “柯” 的Unicode码是 \u 67ef(0110 0111 1110 1111)，根据上表，我们可以知道 “柯”这个Unicode对应的UTF-8编码需要3个字节。然后，从0110 0111 1110 1111从由往左依次取数放入到1110 xxxx 10xx xxxx 10xx xxxx中，得到最终的结果是 1110 0110 1001 1111 1010 1111，转换成十六进制是E6 9F AF。

linux进程间通信

一、简介

1.1 基本概念

IPC是进程间通信(interprocess communication)的简称。IPC指的是不同进程间消息传递的不同方式。

消息传递历经如下几个发展阶段：

  • 管道。其缺点是：
    • 只能在具有共同祖先（指父子进程关系）的进程间使用
    • 半双工（单向）

  • 有名管道(named pipe)

    • 半双工
    • 可以在没有亲缘关系的进程间使用

  • System V消息队列：20世纪80年代早期加到System V内核中。可以在没有亲缘关系的进程中通信。

  • Posix消息队列：同一主机上游亲缘关系和无亲缘关系的进程之间。

    • 两个消息队列的缺点：存在于内核并且没有引用计数；性能跟全双工管道和套接字相当。
  • 远程过程调用：RPC可用在同一主机上的客户和服务器直接，因此可以认为RPC是另一种形式的消息传递。

1.2 进程、线程与信息共享


Unix进程间的信息共享有多种方式：

  • 文件系统：

    • 两个进程共享存留于文件系统中某个文件上的某些信息。为访问这些信息，每个进程都得穿越内核。

  • 内核：

    • 两个进程共享驻留于内核中的某些信息。管道是共享内核的一个例子，System V消息队列和System V信号灯也是。
  • 共享内存区：
    • 两个进程有一个双方都能访问的共享内存区。每个进程一旦设置好该共享内存区，就能根本不涉及内核而访问其中数据。

1.3 IPC对象的持续性

有三种类型的持续性：

  • 随进程持续
  • 随内核持续
  • 随文件系统持续

1.4 名字空间

当两个或多个无沁园关系的进程使用某种类型的IPC对象类彼此交换信息时，该IPC对象必须有一个某种形式的名字name或标识符identifier，这样其中一个进程可以创建该IPC对象，其余进程可以指定同一个IPC对象。

管道没有名字，因此只能用于有亲缘关系的进程间。

1.5 fork,exec对于IPC对象的影响

二、管道和FIFO

2.3 管道

管道由pipe函数创建，提供一个单路（单向）数据流。

int pipe(int fd[2])

该函数返回两个文件描述字：fd[0]和fd[1]。前者打开来读，后者打开来写。

创建一个全双工IPC管道的另外一个方法是使用UNPv1的14.3节讲述的socketpair函数。

当然管道最常见用途是用在各种shell中，这种情况下半双工就够了。

2.4 全双工管道

2.6 FIFO 有名管道

FIFO指代先进先出(first in, first out)，是一个单向（半双工）数据流。

与管道不同的是， 每个FIFO有一个路径与之关联，从而允许无亲缘关系的进程访问同一个FIFO。FIFO也称为有名管道（named pipe)。

因此FIFO可以用于无亲缘关系的进程间通信。

通过int mkfifo(const char *pathname, mode_t mode)创建。

三、Posix 消息队列

3.1 简介

消息队列可以认为一个消息链表。每个消息是一个记录，它由发送者赋予一个优先级。在某个进程往一个队列写入消息之前，并不需要另外某个进程在该队列上等待消息的到达。

队列中的每个消息具有如下属性:

  • 一个无符号整数优先级或一个长整型类型
  • 消息的数据部分长度
  • 数据本身

消息队列与管道和FIFO区别：

  1. 某个进程往一个队列写入消息之前，并不需要某个进程在该队列上等待消息的到达；而管道和FIFO 必须是读出者已经存在，否则先有写入者没有意义。
  2. 管道和FIFO是字节流模型，没有消息边界，也没能与每个消息关联的类型。

3.6 Posix消息队列的 mq_notify函数

System V消息队列的问题之一是无法通知一个进程何时在某个队列中放置了一个消息。

Posix消息队列允许异步事件通知（asynchronous event notification),以告知何时有一个消息放置到了某个空消息队列中，这种通知有两种方式可供选择：

  • 产生一个信号
  • 创建一个线程执行一个指定函数

3.8 使用内存映射I/O实现Posix消息队列

该小节使用内存映射I/O 以及一个Posix互斥锁和一个Posix条件变量以约500行的C代码实现了Posix消息队列的大多数特性。

四、System V消息队列

System V消息队列使用消息队列标识符标识。

新的应用程序应考虑使用Posix消息队列。

Posix消息队列遗漏的主要特性是从队列中读出指定优先级的消息的能力。这两种消息队列都不使用真正的描述字，从而造成在消息队列上使用select或poll的困难。

####消息队列和全双工管道对比

unp15.7节中介绍了二者对比。

如果需要客户进程和服务器进程之间的双向数据流，可以使用消息队列或全双工通道。（通过Unix域套接字机制 或者pipe函数提供全双工管道）。

图15-27 显示了在Solaris上3中技术在时间方面的比较，这三种技术是：消息队列、全双工管道、UNIX域套接字。

测试程序先创建IPC通道，调用fork，然后从父进程向子进程发送约200MB数据。数据发送的方式是：对于消息队列，调用100,000次msgsnd，每个消息长度为2000字节。对于全双工管道和UNIX域套接字，调用100,000次write，每次写2000字节。时间都以秒为单位。

从这些数字中可见，消息队列原来的实施目的是提供高于一般速度的IPC，但现在与其他形式的IPC相比，在速度方面已经没有什么差别了。

注意：在原来实施消息队列时，可用的其他形式的IPC就只有半双工管道这一种。

考虑到消息队列遇到的问题，我们得出的结论是，在新的应用程序中不应当再使用它们。

消息队列遇到的问题是：

  1. 消息队列是在内核中起作用，没有引用计数。例如，如果进程创建了一个消息队列并写入了几则消息，然后终止，那么该消息队列及其内容不会被删除。
  2. 消息队列在文件系统中没有名字。为了支持这些IPC对象，内核中增加了十几个全新的系统调用(msgget,semop,shmat等）。

五、总结

管道和FIFO：

  • 半双工管道
  • 管道是匿名的、FIFO是有名的；因此管道只能用于有亲缘关系的进程、FIFO则可以用于没有亲缘关系的进程。
  • 对于shell而言使用管道即可。

消息队列：

  • 诞生的背景是IPC只有管道，因此通过消息队列提升IPC性能；但后来有了全双工管道和套接字，
  • 因此新的应用程序都可以使用全双工管道和套接字。
  • 消息队列缺点：存在于内核并且没有引用计数，会存在进程已经消失，但是消息队列还在内核中存在。

套接字：

  • 全双工通道
  • 应用场景：
    • 在nginx master进程和worker进程之间是通过套接字进行通信的。
    • 在nginx和php-fpm之间通过套接字进行通信。

5.1 nginx和phpfpm交互两种方式性能比较

nginx和phpfpm的交互有两种方式：

  • tcp方式
  • unix domain socket方式
    tcp方式的优点是可以跨机器。

socket方式的优点是：

  • 进程间通信，
  • 不需要经过网络协议栈、不需要打包、拆包、计算校验和、维护序号和应答等，只是将应用层数据从一个进程拷贝到另外一个进程。
  • 因此IPC机制本质上是可靠的通讯，而网络协议是为不可靠通讯设计的。

5.2 IPC分类

  1. 消息传递（管道、FIFO、消息队列、RPC、匿名共享内存区、有名共享内存区）
  2. 同步（互斥锁、条件变量、读写锁、文件与记录锁、信号灯）

golang包导入

go源代码是按package方式组织，再通过import引入使用。

工作目录

在Go中代码保持在称之为workspace的系统文件夹中。这个工作目录有三个根目录：

  • bin：包含可执行文件
  • pkg：包含不同操作系统架构的包二进制文件。
  • src：包含按包方式组织的源代码

其中bin和pkg文件夹是在调用go命令安装和编译源代码时自动生成。

必须让Go知道工作目录的位置，这样才能知道包的具体位置。 通过设置环境变量GOPATH来指定。

导入包

  1. $GOPATH/src/importpackage/lib/lib.go

    package lib

import "fmt"

func SayHello() {
    fmt.Println("Hello,I'm in myLib :) ")
}

  2. $GOPATH/src/importpackage/app/main.go

    package main

import "importpackage/lib"

func main() {
    lib.SayHello()
}

  3. 目录结构：

    .
└── src
    └── importpackage
        ├── app
        │   └── main.go
        └── lib
            └── lib.go

    go build -o main src/importpackage/app/main.go

导入包的多种方式

  • 代码统一存储在工作目录下
  • 工作目录里面有多个包，不同包按目录组织，包下面由多个代码文件组成。
  • 导入包时按包的唯一路径进行导入，导入的包默认是必须要使用，如果不使用则编译失败，需要移除，减少不必要代码的引入，当然还有其他使用场景。默认情况下，我们使用文件名做为包名，方便理解。不同包组织不同的功能实现，方便理解。
