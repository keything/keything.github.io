2019-08-27C标准IO库与UnbufferIO库

C标准I/O库函数如何用系统调用实现的

fopen

调用open打开指定的档案，返回一个文档描述符，分配一个FILE
结构体，其中包含该档案的描述符、IO缓冲区、当前读写位置等信息，返回这个FILE结构体的地址。

fgetc

通过传入的FILE*参数找到该档案的描述符、I/O缓冲区和当前读写位置，判断能否从I/O缓冲区中读到下一个字，如果能够读到就直接返回；否则调用read，把文档描述符传进去，让内核读取该档案的数据到I/O缓冲区，然后返回下一个字。

对于C标准I/O库来讲，打开的档案由FILE*指针标识；对于内核来讲，打开的档案由文档描述符标识，文档描述符从open系统调用获得。

fputc

判断该档案的I/O缓冲区是否有空间再存放一个字，如果有空间则直接保存在I/O缓冲区并返回；如果已经满了，则调用write，让内核把I/O缓冲区内容写到文件。

库函数与系统调用的层次关系

open,read,write,close等系统函数称之为无缓冲I/O（unbuffered I/O)函数，因为它们位于C标准库的I/O缓冲区的底层。

缓冲和无缓冲对比

  • 用 unbuffered I/O函数每次读写都要写进内核。调用一个系统调用比一个用户空间的函数要慢很多；所以在用户空间打开I/O缓冲区还是有必要的。使用标准I/O库函数，省去了自己管理I/O缓冲区的麻烦。
  • 用C标准I/O函数要时刻注意I/O缓冲区与实际档案不一致，在必要时需要调用fflush
  • 网络编程语言通常直接调用Unbuffered I/O函数
  • buffered I/O
      • 读取：硬盘—>内核缓冲区—>用户缓冲区—>用户程序
      • 写回：用户程序—>用户缓冲区—>内核缓冲区—>硬盘

  • unbuffered I/O

      • 读取：硬盘—>内核缓冲区—>用户程序
      • 写回：用户程序—>内核缓冲区—>硬盘

    • Linux内核中存在页高速缓存（page cache），read时，若数据在页高速缓存中，则直接从页高速缓存中读取，若数据不在页高速缓存中，则内核将数据从磁盘上加载到页高速缓存中，然后再从页高速缓存中读取；write时，Linux采用write-back策略，程序执行写操作直接写到缓存中，并不直接将数据同步到磁盘上，而是将页高速缓存中被写入的页面标记为dirty，并且被加入dirty page链表中，然后由一个进程（回写进程）周期性将dirty page链表中的页写回到磁盘。

  • direct I/O

    • direct Io 才是真正的什么缓冲区都没有，直接与硬盘交互
    • 读取：硬盘—>用户程序
    • 写回：用户程序—>硬盘

glog中几个重要的参数配置

src/windows/log/logging.h 进行声明

参数名称 作用 默认值
logtostderr 只输出到stderr，不输出到文件 false
alsologtostderr 既输出到stderr，又输出到文件 false
stderrthreshold 日志消息等级大于等于 配置的阈值，则自动既输出到stderr，又输出到文件 默认值是 ERROR，大于等于ERROR（即ERROR和FATAL都会输出到stderr和file
logbuflevel 日志等级小于等于这个阈值的，才被缓存。默认取值为0，只有INFO被缓存；取值为-1时，则任何都不被缓存 0
logbufsecs 最多缓存日志的秒数 默认是30秒
minloglevel 小于这个等级的日志不会被打印 默认是0，即INFO
log_dir 日志目录 默认是从环境变量GOOGLE_LOG_DIR
stop_logging_if_full_disk 当磁盘满的时候停止写日志 false

在实际使用中遇到的几个问题：

1，锁竞争。 stderrthreshold和logbuflevel都是默认值，因此

  • LOG(ERROR)时 不会走入buffer，而是直接flush；同时还会输出到stderr
  • LOG(WARNING)时，不会走入buffer，而是直接flush；但不会输出到stderr

因此LOG(ERROR）带来了更严重的锁竞争，LOG(WARNING)则稍微好一点，但由于直接flush时间很长，一次日志写入占有锁的时间就会变长，导致性能变差。

解决办法：

1，为了避免大家在后续使用中再出现类似问题，因此可以做出以下改变：

  • logbuflevel 更改为 ERROR等级，这样INFO，WARNING，ERROR都会被缓存
  • stderrthreshold 更改为FATAL等级，因此ERROR也不会被输出到stderr.

2019-08-24系统分析

一、概述

系统性能分为on-cpu和off-cpu两种，通过分析这两种情况才能定位到系统性能问题到底是什么。

  • on-cpu：
    程序运行在CPU的时间，在所有代码路径上的分布

  • off-cpu:
    进程不运行在任何CPU上的时间。一般是这个进程因为某个原因处于休眠状态。

    • off-cpu原因：
      • 1， 与当前thread有关：
        • I/O
        • blocks
        • mutex 比如等待某一个系统级别的锁
      • 2, 与当前thread无关
        • interrupt 中断 （可以通过 top 按1，查看si(soft interrupt),hi(high interrupt))
        • CPU高需求的上下文切换，被一个非常繁忙的进程调度器scheduler强行剥夺CPU时间片
        • paging/swapping

蓝色部分是off-cpu。

我们在追查问题的时候，推荐的策略是一种”小步推进，连续发问”的方式。我们会把最终问题的假设分解成一系列的小假设，然后逐步求索，验证。

二、介绍

on-cpu分析和off-cpu分析都有采样和跟踪(sampling and tracing)两种方式。

2.1 采样 sampling

on-cpu的采样是运行在cpu上的时候进行采样；off-cpu的采样是off-cpu的时候进行采样；

但是系统profile很少使用off-cpu 采样。 原因是什么？

通过每个cpu的计时器中断(per-CPU timer interrupts) 来实现采样，之后再去分析当前被中断的正在运行的进程：产出一个on-cpu profile。

off-cpu则是一个完全不同的方式，有两种方式：

  • 每个应用线程中创建一个计时器，将应用线程唤醒并采集栈
  • 在一个间隙内核遍历所有线程并采集栈

使用cpu sampling可以很高效的分析on-cpu问题，包括 热点代码路径和自适应自旋锁（hot code-paths and adaptive mutex spins)，当应用被阻塞并处于off-cpu时，不采集数据。

记录函数执行的开始和结束的时间戳，从而函数花费的时间可以被计算出来。如果时间戳包含了流逝的时间和CPU时间，那么就有可能计算出在CPU上计算慢的函数，以及 哪些函数是因为被阻塞在off-cpu上导致变慢的。

与sampling不同，这些时间戳的精度很高。

当然有一个缺点，你可以追踪所有应用函数，这将会带来严重的性能影响，或者 可以选出被阻塞的函数。

2.2.2 off-cpu tracing

只有将线程切换到off-cpu的内核函数才会被追踪，时间戳和用户态栈也会被追踪。
这聚焦于off-cpu事件，不需要跟踪所有的应用函数，也不需要知道应用程序是什么。 这个方法可以用于分析任何blocking事件。

off-cpu tracing抓取任何应用程序的所有等待事件。

分析off-cpu使用的主要方法就是off-cpu tracing。

3. Linux: perf, eBPF

在linux上进行off-cpu分析的tracer有很多。这儿我使用eBPF，很容易进行栈追和时间的内容总结。eBPF是内核的一部分，可以使用bcc工具。但这要求内核必须是4.8以上。

  • 内核4.8及其以上版本 可以使用eBPF进行分析
  • 内核4.8以下版本， 有不同的方法去分析每一种阻塞类型：
    • storage tracing for storage I/O
    • kernel statistics for scheduler latency
    • 使用SystemTap 和 perf event logging。

4. 警告

最大的警告是off-cpu分析的开支。需要意识到有scheduler latency 和 involuntary context switches。

4.1 Scheduler Latency

Something that’s missing from these stacks is if the off-CPU time includes time spent waiting on a CPU run queue. This time is known as scheduler latency, run queue latency, or dispatcher queue latency. If the CPUs are running at saturation, then any time a thread blocks, it may endure additional time waiting its turn on a CPU after being woken up. That time will be included in the off-CPU time.

You can use extra trace events to tease apart off-CPU time into time blocked vs scheduler latency, but in practice, CPU saturation is pretty easy to spot, so you are unlikely to be spending much time studying off-CPU time when you have a known CPU saturation issue to deal with.

4.2 Involuntary Context Switching

If you see user-level stack traces that don’t make sense – that show no reason to be blocking and going off-CPU – it could be due to involuntary context switching. This often happens when the CPUs are saturated, and the kernel CPU scheduler gives threads turns on CPU, then kicks them off when they reach their time slice. The threads can be kicked off anytime, such as in the middle of a CPU heavy code-path, and the resulting off-CPU stack trace makes no sense.

golang性能追查的几个case

0，概述

  • cpu变化不大，但时延变化过大时，可能是由于锁、阻塞等off-cpu的问题造成
  • 短连接会影响性能，即多花费三次握手的时间，不会造成非常大的性能问题；如果造成非常大的性能问题，可以考虑是不是有其他原因
  • 系统一个资源被耗尽时，那么cpu idle、时延等都会出现垂直变化
  • golang中json序列化使用陶文json-iterator库 比原生的库，cpu使用率可以下降3%（700qps）

  • 异步：间接的方式影响。 An example of this is a file system which buffers writes and flushes them to disk later. The eventual I/O and their latency may be high, but not endured directly by the application. They may be endured indirectly, as contention with later synchronous reads.

Off-CPU 分析

性能问题可以归结为两类：

  • On-CPU: 在CPU上进行计算，耗时发生在CPU上
  • Off-CPU: IO阻塞、锁、计时器、页交换（paging/swapping)等

Off-CPU分析是一个性能方法论，off-CPU时间可以与上下文如堆栈等 一起 被测量和研究。
与CPU性能分析不同，CPU性能分析只看在CPU上执行的线程。 Off-CPU的目标是 线程状态阻塞并且处于Off-CPU。正如右图中蓝色所展示的部分。 Generice thread states

通过Off-CPU分析和CPU分析，可以理解线程的100%的时间。 与追踪技术不同，这个方法的目标是理解阻塞的核心调度的概念，能够方便的捕获所有。（Off-CPU analysis is complementary to CPU analysis, so that 100% of thread time can be understood. This method also differs from tracing techniques that instrument applications functions that commonly block, since this method targets the kernel scheduler’s notion of blocking, which is a convenient catch-all.）

线程离开CPU有很多原因，包括I/O，locks，但也有很多不相关的线程操作，包括 由于对CPU资源高要求的上下文切换，和中断。不管什么原因，在一个请求中发生了，就会引入延时。

Go 1.11 Modules

译文

Go1.11 中引入了之前提到的版本模块(versioned modules)。modules在Go 1.11版本中是实验的性质，计划在Go 1.13中引入最终版本。尽管还有很多细节会改变，但未来版本都会支持Go1.11 或Go1.12中定义的模块。

2018年1月提出了原型vgo。2018年7月 模块登录主仓库。Go1.1在2018正式发布。

目录内容

快速开始

先看一下如何使用，后面再看一下细节。

  1. 在GOPATH目录外创建一个目录

    $ mkdir -p /tmp/scratchpad/hello
    $ cd /tmp/scratchpad/hello

  2. 初始化一个新的模块

    go mod init github.com/you/hello

  3. 编写代码

    $ cat < hello.go
    package main

    import (

    "fmt"
"rsc.io/quote"

    )

    func main() {

    fmt.Println(quote.Hello())

    }
    EOF

  4. 执行

    $ go build
    $ ./hello

go.mod文件会自动更新，明确指定你的依赖的版本，v1.5.2是一个semver 标记:

cat go.mod

日常工作方式

在上面的例子中不需要使用 go get。
你的日常工作方式如下：

  • 在 .go代码中添加你的语句。

New Concepts

Modules

Module是相关go package的集合，将这些package聚合在一起作为一个独立单元。

Module记录精确的依赖，并创建可复用的构建。

通常，一个版本控制的仓库在其根目录准确的包含一个模块。当然在一个仓库中也支持多个模块。

总结仓库、模块、包的关系：

  • A repository contains one or more Go modules.
  • Each module contains one or more Go packages.
  • Each package consists of one or more Go source files in a single directory

模块必须是根据semver进行语义版本化的，其格式是v(major).(minor).(patch)。比如v0.1.0,v1.2.3,v1.5.0-rc.1。 开头的v是必须的。 如果使用Git，tag就是版本。

go.mod

模块是由go.mod文件和go 源码的文件树定义的。 模块源码需要位于GOPATH目录之外。有四个指令：module,require,replace,exclude.

下面是一个定义模块github.com/my/thing的go.mod文件的例子，

module github.com/my/thing

require (
    github.com/some/dependency v1.2.3
    github.com/another/dependency/v4 v4.0.0
)

通过module指令表名模块的身份，并提供了module path。引入模块内所有包的话，都使用包的path 作为前缀。 模块路径和包相对路径决定了包的import path。

例如，创建了模块github.com/my/repo，模块由两个包组成 github.com/my/repo/foo 和 github.com/my/repo/bar组成。go.mod第一行通常声明模块路径，即 module github.com/my/repo，对应到硬盘的目录结构是：

repo/
├── go.mod
├── bar
│   └── bar.go
└── foo
    └── foo.go

在Go源码中，使用包含模块路径的全路径引入包。举例，go.mod文件中声明模块路径是example.com/my/module，一个消费方应该是这样使用:

import "example.com/my/module/mypkg

这就是从模块example.com/my/module中引入包mypkg的做法。

版本选择(version selection)

  • 引入go.mod中不存在的包时，go build和go test命令都会自动找到合适的模块 和 最大版本的依赖，写入到go.mod文件 作为一个require指令。
  • 一个构建中，所有模块的版本选择使用最小版本选择(minimal version selection)算法。对于构造中的每个模块而言，使用的是require指令中明确列出的语义最大化的版本。

举个例子，如果你的模块依赖模块A 需要D v1.0.0，你的模块同时依赖模块B 需要D v1.1.1，那么根据最小版本选择，那么D v1.1.1会被选择用于构建。即使D v1.2.0可用了，也依然使用D v1.1.1。

最小版本选择算法的简短介绍可以看High Fidelity Builds 或者看 more detailed vgo blog series

语义版本化（Semantic Import Versioning)

这么多年以来，Go官方FAQ中关于包版本有这些建议：

"Packages intended for public use should try to maintain backwards compatibility as they evolve. The Go 1 compatibility guidelines are a good reference here: don't remove exported names, encourage tagged composite literals, and so on. If different functionality is required, add a new name instead of changing an old one. If a complete break is required, create a new package with a new import path."

最后一句很重要：如果破坏了兼容性，你就要更改包的引用路径。 在Go1.11 模块中，这些建议被写入import compatibility rule中：

"If an old package and a new package have the same import path, the new package must be backwards compatible with the old package."

基于Semantic Importing Versioning，Go模块中遵循下面规则：

  • 遵从semver
  • 如果模块是v2或更高版本，那么引用路径要加上版本号。如 github.com/my/mode/v2/
  • 如果版本是v0或v1，则不需要在引用路径上加上版本。

linux同步机制

日常学习和工作中经常用到linux同步机制，因此进行一个总结。

一、为什么需要同步

很简单，同一个资源的修改只能同时被一个进程或线程修改，不然会出现异常情况。

二、同步的机制

  1. 原子操作
  2. 自旋锁
  3. 信号量
  4. 互斥锁
  5. 读写锁

三、自旋锁

自旋锁是linux内核中使用非常频繁的锁，而我们在写c++或golang代码时基本不会直接使用自旋锁。 但是我们遇到性能问题时，通过perf top去查看时经常会看到spin_lock占了非常多的CPU，这是因为很多锁的底层实现都会使用自旋锁。

所谓自旋锁：

当进程A申请锁成功后，进程B申请锁就会失败，但是进程B不会调度，原地自旋。就在原地转到天昏地暗只为等到进程A释放锁。由于不会睡眠和调度的特性，在中断上下文中，数据的保护一般都是选择自旋锁。

自旋锁的实现：

linux中针对每一个spin lock会有两个计数。分别是next和owner（初始值为0）。进程A申请锁时，会判断next和owner的值是否相等。如果相等就代表锁可以申请成功，否则原地自旋。直到owner和next的值相等才会退出自旋。假设进程A申请锁成功，然后会next加1。此时owner值为0，next值为1。进程B也申请锁，保存next得值到局部变量tmp（tmp = 1）中。由于next和owner值不相等，因此原地自旋读取owner的值，判断owner和tmp是否相等，直到相等退出自旋状态。当然next的值还是加1，变成2。进程A释放锁，此时会将owner的值加1，那么此时B进程的owner和tmp的值都是1，因此B进程获得锁。当B进程释放锁后，同样会将owner的值加1。最后owner和next都等于2，代表没有进程持有锁。next就是一个记录申请锁的次数，而owner是持有锁进程的计数值。

共享数据如果是被进程上下文使用，可以使用信号量和互斥量。
共享数据如果是被中断上下文使用，则必须使用自旋锁。 因为在中断上下文中不能使用睡眠的锁。 信号量和互斥量都是睡眠锁。

Spinlock的特点

我们可以总结spin lock的特点如下：

（1）spin lock是一种死等的锁机制。当发生访问资源冲突的时候，可以有两个选择：一个是死等，一个是挂起当前进程，调度其他进程执行。spin lock是一种死等的机制，当前的执行thread会不断的重新尝试直到获取锁进入临界区。

（2）只允许一个thread进入。semaphore可以允许多个thread进入，spin lock不行，一次只能有一个thread获取锁并进入临界区，其他的thread都是在门口不断的尝试。

（3）执行时间短。由于spin lock死等这种特性，因此它使用在那些代码不是非常复杂的临界区（当然也不能太简单，否则使用原子操作或者其他适用简单场景的同步机制就OK了），如果临界区执行时间太长，那么不断在临界区门口“死等”的那些thread是多么的浪费CPU啊（当然，现代CPU的设计都会考虑同步原语的实现，例如ARM提供了WFE和SEV这样的类似指令，避免CPU进入busy loop的悲惨境地）

（4）可以在中断上下文执行。由于不睡眠，因此spin lock可以在中断上下文中适用

场景分析

线程A访问共享资源R，线程B访问共享资源R；

  1. 没有任何锁机制，
    当线程A正在访问R 进行写操作时，线程B访问R进行写操作，同一个资源的同时写操作，会导致程序执行不正确。

  2. 加入自旋锁
    线程A 先成功获得自旋锁，对R进行写操作。 此时B不能获得自旋锁，则一直在等待，线程B不能做其他事情，只是在等待自旋锁。

如果线程A在对R进行写操作时，现在发生中断并执行的是线程B，那么线程B等待自旋锁会一直在等待；而A由于发生中断又不会释放锁，因此出现死锁 。

为了解决这样的问题，linux kernel采用了这样的办法：如果涉及到中断上下文的访问，spin lock需要和禁止本CPU上的中断联合使用。

四、信号量

信号量（semaphore）是进程间通信处理同步互斥的机制。是在多线程环境下使用的一种措施，它负责协调各个进程，以保证他们能够正确、合理的使用公共资源。 它和spin lock最大的不同之处就是：无法获取信号量的进程可以睡眠，因此会导致系统调度。

信号量实现：

一个count计数：标记当前可用资源数量
一个链表：记录等待的进程

五、读写锁

不管是自旋锁、信号量、还是互斥锁 在同一时间只能有一个进程进入临界区。对于有些情况，我们是可以区分读写操作的。因此，我们希望对于读操作的进程可以并发进行。对于写操作只限于一个进程进入临界区。而这种同步机制就是读写锁。读写锁一般具有以下几种性质：

  • 同一时间有且仅有一个写进程进入临界区。
  • 在没有写进程进入临界区的时候，同时可以有多个读进程进入临界区。
  • 读进程和写进程不可以同时进入临界区。

对于某个给定资源的共享访问也称为共享-独占（shared-exclusive)上锁，因为获取一个读写锁用于读称为共享锁（shared lock)；获取一个读写锁用于写称为独占锁（exclusive lock）。

六、互斥锁

前文提到的semaphore在初始化count计数的时候，可以分为计数信号量和互斥信号量（二值信号量）。mutex和初始化计数为1的二值信号量有很大的相似之处。他们都可以用做资源互斥。但是mutex却有一个特殊的地方：只有持锁者才能解锁。但是，二值信号量却可以在一个进程中获取信号量，在另一个进程中释放信号量。如果是应用在嵌入式应用的RTOS，针对mutex的实现还会考虑优先级反转问题。

互斥锁指代相互排斥(mutual exclusion)，是最基本的同步形式，用于保护临界区(critical region)，以保证任何时刻只有一个线程在执行其中代码。保护一个临界区的通常轮廓是：

lock_the_mutex(…)
临界区
unlock_the_mutex(…)

互斥锁保护的是临界区，实际上保护的是在临界区中被操纵的数据（data）。也就是说，互斥锁通常用于保护由多个线程分享的共享数据（shared data）

6.1 生产者和消费者问题

在UNP 第7章中讲述了典型的生产者和消费者的问题。

场景一：生产者全部生产以后，消费者再消费

UNP给出的实现是多生产者单消费者的方式，因此生产者需要对buff加互斥锁。而消费者只需要顺序消费buff中内容即可。

场景二：边生产边消费

那么消费者所在线程会有两种情况：一种是buff内有数据，直接消费；第二种是buff内没有数据，那么消费者线程就必须一直等待，直到有数据到来，才开始消费。

UNP中给出了一个实现（UNP中给出的是一个函数，我是直接写的）：

while(true) {
lock_the_mutex(..)
if (buf.empty) {
unlock_the_mutex(…)
continue
} else {
unlock_the_mutex(…)
break
}
}
operate buf data

上面例子中，当期待的数据没有准备好时，我们的做法是：一次次循环，每次给互斥锁解锁、上锁。这称为轮询（polling），是一种对CPU时间的浪费。

当然我们也许可以睡眠很短的一段时间，但是不知道应该睡眠多久，这儿需要的是另外一种类型的同步，它允许一个线程（或进程）睡眠到发生某个事件为止。

七、条件变量

互斥锁用于上锁，条件变量用于等待。

那么在消费者进程中实现可以变为：

pthread_mutex_lock(mutex_)
while (ready == 0) {
pthread_cond_wait(cond,mutex)
}
pthread_mutex_unlock(mutex_)

在生产者进程中实现变为

pthread_mutex_lock(mutex_)
pthread_cond_signal(cond_)
pthread_mutex_unlock(mutex_)
